# Onboarding-Tour (Portal) — Brief

Single source of truth for the interactive portal product-tour. Re-read this first in any resumed session.

## Goal

Ship an **interactive, on-brand guided tour** inside the portal that **drives the real UI** (navigates routes, switches sub-tabs, spotlights real on-screen elements with explainer cards) to **showcase the full value of the portal** to a new Inhaber. Auto-launches once on first login, fully skippable, and **re-launchable from Einstellungen**.

## Success criteria

- **Value is seen.** A first-login Inhaber who completes the core tour has been walked past every major area and told, in plain Sie-German, what EINS does for them in each. The point is perceived value, not feature-counting.
- **Real navigation works.** The tour reliably navigates dashboard to Anfragen to a Bewertungen sub-tab etc., **waiting for each target to mount** before spotlighting. No empty or misplaced spotlights, no flicker.
- **Never nags.** Skippable at any step. Once completed or dismissed it **never auto-launches again** (persisted per user). Re-launchable anytime from an Einstellungen card.
- **Tiered.** One short core showcase tour everyone sees, plus **optional deep-dive chapters** per area, launched on demand.
- **On brand.** Mint accent, EINS card styling, sentence case, formal Sie, no all-caps, no em-dashes, no corny tells. Motion is smooth, not janky.
- **Empty-state safe.** Copy never assumes data exists; the tour works on a fresh account.

## Constraints

- **Scope:** portal app only (`apps/portal`), **Inhaber-first**. Next.js 14 App Router, React 19.
- **One new runtime dep only: `driver.js`** (~5kb, zero deps; matches the "lightweight" decision). **No Framer Motion** (the portal deliberately uses CSS transitions; keep deps minimal). Animation via Tailwind/CSS + driver.js's own spotlight transitions.
- **Copy rules** (CLAUDE.md): formal Sie, no em-dashes (U+2014), no all-caps / no `tracking-wider`, sentence case, "Praxis" not "Klinik", anti-Anglicism mappings, no corny tells ("nicht X sondern Y", announced honesty, etc.).
- **Tabs are routes, not Radix.** Bewertungen and Leitfaden "tabs" are `<Link>` navigations. "Switching a tab" = a route push; the controller must handle the async mount gap.
- **Persistence:** per-user flags on `clinic_users` (new columns, **migration 0063**). Mirror the existing server-action + `writeAudit` pattern from `einstellungen/actions.ts`.
- **Do not duplicate** the existing `/onboarding` "Erste Schritte" checklist. The tour complements it and can point at it.
- **Avatar:** non-technical owners 40-65. Product success means needing the portal *less* over time, so the tour must be fast, skippable, value-framed, and never patronizing.

## Non-goals (v1)

- **No Marketing/Frontdesk tours.** Inhaber only for v1 (a 3-step Frontdesk "Anfragen" tour is a later candidate).
- **No XP / badges / levels / leaderboards.** Gamification = a "Schritt x von y" progress indicator, smooth spotlight motion, and one tasteful finish moment. Brand copy rules win over "video-game".
- **No non-dev CMS** for tour content. Steps live in code (a typed step config).
- **No analytics funnel dashboard.** Completion is recorded via the per-user flag + audit log only.
- **No new i18n.** German only.
- **Not** rebuilding the Erste-Schritte checklist, Fragebogen, or Checkliste.

## Key open assumptions

- **Auto-launch trigger:** on a portal load where `role = inhaber` and both `onboarding_tour_completed_at` and `onboarding_tour_dismissed_at` are null, show a small "Willkommen. Rundgang starten?" prompt (Starten / Später). It does not force itself full-screen.
- **Anchors:** spotlight targets get explicit `data-tour="..."` attributes (nav items, dashboard tiles, key tabs, settings cards) so selectors stay stable through refactors. Nav already has per-href refs we can extend.
- **Mobile:** the tour opens the nav drawer when a nav step needs it. If cross-route mobile spotlighting proves fragile, v1 may gate **auto-launch to desktop** and show a "am besten am Computer" note on mobile, while still allowing manual start. Decision deferred to the relevant bucket.
- **Deep-dive chapters (v1 set):** Anfragen-Detail, Werbebudget, Bewertungen, Medien. Launchable on demand (per-page entry or a small hub). Exact set finalized in the content bucket.
- **Mount point:** a client `TourProvider` mounted inside the server `(portal)/layout.tsx`, wrapping `PortalShell`, fed `role` + the two persistence flags from the existing session fetch.
